# /tdc — TechDog Claude

멀티 에이전트 개발 오케스트레이션의 메인 진입점입니다.

## 사용법

```
/tdc <파일.md>              스펙 파일을 읽고 기획 → 개발까지 진행
/tdc <설명>                 텍스트로 간단히 지시
/tdc plan <파일.md|설명>    기획만 진행
/tdc dev <파일.md|설명>     개발만 진행
/tdc debug <설명>           디버깅
/tdc review [파일들]        코드 리뷰
/tdc session <명령>         세션 관리
```

## 실행 흐름

### 1단계: 입력 분석

사용자 입력을 다음 순서로 분석합니다:

**A. 파일이 전달된 경우** (`.md` 확장자 또는 파일 경로)
1. 해당 파일을 Read 도구로 읽는다
2. 파일 내용을 분석하여 스펙/기획서인지 판단한다
3. 스펙이면 → **기획 워크플로우**로 진입
4. 이미 태스크가 분해된 플랜이면 → **개발 워크플로우**로 진입

**B. 서브커맨드가 있는 경우** (plan, dev, debug, review, session)
- 해당 워크플로우로 직접 라우팅

**C. 텍스트만 전달된 경우**
- 의도를 분석하여 적절한 에이전트 선택
- 단순 질문 → 직접 답변 (에이전트 위임 불필요)
- 구현 요청 → 복잡도에 따라 기획 또는 개발로 라우팅

### 2단계: 프로젝트 초기화

처음 실행 시 자동으로:
```bash
mkdir -p .tdc/{sessions,context,plans}
```

### 3단계: 에이전트 실행

스펙 파일 기반 전체 워크플로우:

```
스펙 파일 읽기
    ↓
Planner Agent — 태스크 분해, 플랜 생성
    ↓ (.tdc/plans/에 저장)
사용자 승인 확인
    ↓ 승인
Developer Agent — 태스크별 순차/병렬 구현
    ↓
Reviewer Agent — 자동 코드 리뷰
    ↓
결과 보고
```

## 스펙 파일 형식

사용자가 작성하는 스펙 파일은 자유 형식이지만, 다음 정보가 포함되면 좋습니다:

```markdown
# 프로젝트 이름

## 목적
무엇을 왜 만드는지

## 기술 스택
사용할 언어, 프레임워크, DB 등

## 기능 목록
- 기능 1
- 기능 2

## 기타 요구사항 (선택)
- 성능, 보안, 테스트 등
```

**참고:** 빈 스펙 템플릿이 필요하면 `/tdc init`을 실행하세요. `spec-template.md`가 생성됩니다.

## 토큰 최적화 규칙

- 스펙 파일 내용은 planner에게만 전달 (다른 에이전트에는 플랜만 전달)
- 단순 작업은 에이전트 위임 없이 직접 처리
- 각 에이전트에게는 해당 태스크에 필요한 최소 컨텍스트만 전달
