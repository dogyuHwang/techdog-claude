# /tdc-learn — Project Learning

기존 프로젝트의 기술 스택·구조·컨벤션을 자동 분석해 `.tdc/project-memory.md`에 저장합니다.
**이후 모든 tdc 세션이 이 지식을 활용하여 더 나은 결과를 냅니다.**

## 기본 동작 (`/tdc-learn`)

### 1단계: 기존 상태 확인 (증분 업데이트 지원)

1. `.tdc/project-memory.md`가 이미 존재하면:
   - `# Last updated:` 헤더에서 마지막 학습 타임스탬프를 읽는다
   - `git log --since=<timestamp> --name-only --pretty=format:""` 으로 변경된 파일을 확인
   - 변경 파일이 없으면 → "이미 최신 상태입니다" 출력 후 종료
   - 변경 파일이 있으면 → 기존 파일을 `.tdc/project-memory.md.bak`으로 백업 후 **자동 재분석**
2. 없으면 → 바로 전체 분석 시작

### 2단계: 프로젝트 분석 (3-Layer Deep Scan)

Explore Agent로 3계층 분석을 수행한다:

#### Layer 1: 설정 파일 스캔 (빠른 기술 스택 감지)

**기술 스택 감지** (파일 존재 여부로 판단):
- `package.json` → Node.js/TypeScript (dependencies에서 프레임워크 감지)
- `requirements.txt` / `pyproject.toml` / `setup.py` → Python
- `go.mod` → Go
- `Cargo.toml` → Rust
- `build.gradle` / `pom.xml` → Java/Kotlin
- `pubspec.yaml` → Flutter/Dart
- `next.config.*` → Next.js
- `manage.py` + `settings.py` → Django
- `angular.json` → Angular
- `vite.config.*` → Vite

**코딩 컨벤션 감지**:
- `.eslintrc*` / `.prettierrc*` → 린터/포매터 규칙
- `.editorconfig` → 탭/스페이스, 줄바꿈
- `tsconfig.json` → strict 여부 등

**프로젝트 구조 감지**:
- 최상위 디렉토리 레이아웃
- 엔트리 포인트 (`main.*`, `index.*`, `app.*`)
- 테스트 디렉토리 + 프레임워크
- CI/CD (`.github/workflows/`, `Jenkinsfile`, `.gitlab-ci.yml`)

**빌드/실행 명령 감지**:
- `package.json` scripts 섹션
- `Makefile` 타겟
- `docker-compose.yml` 서비스

#### Layer 2: 실제 코드 샘플링 (Deep Analysis)

Layer 1에서 감지한 주요 디렉토리에서 **디렉토리당 1~2개 파일**을 실제로 읽어 패턴을 추출한다:

**추출 대상 패턴:**
- **Import 스타일**: 절대경로 vs 상대경로, alias (`@/`) 사용 여부, barrel file(`index.ts`) 패턴
- **함수/클래스 패턴**: arrow function vs function declaration, class vs functional
- **에러 처리 패턴**: try/catch vs Result 타입 vs throws, 에러 래핑 방식
- **비동기 패턴**: async/await vs Promise chain, 병렬 처리 방식
- **주석 스타일**: JSDoc vs 인라인, 언어(한/영)
- **export 스타일**: named vs default, 파일 당 export 수

**테스트 파일 샘플링** (테스트 디렉토리에서 1~2개):
- 테스트 구조: describe/it vs test, beforeEach 패턴
- Mock 방식: jest.mock vs manual mock vs stub
- assertion 스타일: expect vs assert vs should

#### Layer 3: 아키텍처 패턴 감지

디렉토리 구조에서 아키텍처 패턴을 추론한다:

| 패턴 | 감지 기준 |
|------|-----------|
| **Feature-based** | `features/`, `modules/` 디렉토리 존재, 기능별 서브디렉토리 |
| **Layered** | `controllers/`, `services/`, `repositories/` 분리 |
| **MVVM** | `viewmodels/`, `views/`, `models/` 분리 |
| **Clean Architecture** | `domain/`, `data/`, `presentation/` 분리 |
| **Monorepo** | `packages/`, `apps/`, `libs/` 최상위 존재 |

추론된 아키텍처는 Planner context pack에 반영한다 (태스크 분해 방향 안내).

### 3단계: 결과물 생성 (3-Pack Output)

## 서브커맨드: `patterns` (세션 패턴 학습)

현재 세션에서 반복 사용된 문제 해결 패턴을 추출해 재사용 가능한 스킬로 저장합니다.

### `/tdc-learn patterns extract`

**추출 대상:**
- 반복 디버깅 패턴 (에러 → 진단 → 해결)
- 특정 스택에서의 관용구 (예: "React hydration 에러 → useEffect 래핑")
- 프로젝트 특화 규칙 (예: "이 프로젝트에서는 항상 ESM import")
- 반복 코드 패턴 (예: "API 엔드포인트 추가 시 validation middleware")

**품질 게이트** (모두 충족해야 저장):
- 재사용 가능성 — 미래에 다시 쓰일 가능성이 높은가?
- 구체성 — 충분히 구체적인가? (너무 일반적이면 거부)
- 검증됨 — 이 세션에서 실제로 문제를 해결했는가?

**저장 경로**: `.tdc/learned-skills/<name>.md`

```markdown
---
name: <skill-name>
triggers: [keyword1, keyword2, ...]
category: debug|pattern|config|workflow
learned_from: <session_id>
confidence: high|medium
---

## Problem
<어떤 상황에서 이 패턴이 필요한가>

## Solution
<구체적 해결 단계>

## Example
<실제 적용 예시>
```

추출 후 내용을 출력하고 **확인 없이 자동 저장**한다. Log: `[AUTO-LEARN] <name> saved`

### `/tdc-learn patterns list`

`.tdc/learned-skills/*.md` 테이블 출력: Name | Category | Triggers | Confidence | Date

### `/tdc-learn patterns apply <name>`

특정 학습 스킬을 현재 세션에 수동 적용한다. 매칭된 스킬 내용을 작업 컨텍스트에 주입.

### `/tdc-learn patterns clean`

confidence: low 이거나 30일 이상 미사용된 스킬을 정리.

### Auto-Injection (자동 주입)

Master Agent가 파이프라인 시작 시 `.tdc/learned-skills/`를 스캔하여:
1. 현재 태스크 키워드와 각 스킬의 `triggers`를 매칭
2. 매칭된 스킬 내용을 해당 에이전트에 컨텍스트로 전달
3. 자동 주입된 스킬은 `[SKILL-INJECTED] <name>` 태그로 로그 기록

**매칭 규칙:**
- confidence: high → 자동 주입 (우선)
- confidence: medium → 자동 주입 (high가 3개 미만인 경우에 한해 추가)
- 한 번에 최대 3개 주입 (토큰 절약)

## 토큰 최적화

- Explore Agent는 파일 존재 여부 위주 빠른 스캔 (내용은 최소)
- 설정 파일은 핵심 필드만 추출
- 패턴 스킬 파일은 200줄 미만 유지
- 자동 주입 시 Problem + Solution만 전달 (Example 생략)
